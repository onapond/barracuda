# Deployment Rules

이 문서는 Barracuda 프로젝트의 빌드/배포 운영 규칙을 명시한다.

목적:
- Vercel 원격 빌드 비용을 피한다.
- 배포 전 검증을 로컬에서 끝낸다.
- 배포 절차를 고정해 작업자별 편차를 줄인다.

---

## 1. 핵심 원칙

- 빌드는 반드시 로컬에서 수행한다.
- Vercel은 배포 용도로만 사용한다.
- 배포 전에 로컬 검증을 완료한다.
- 원격 빌드가 발생하는 기본 `vercel deploy` 흐름은 지양한다.
- 가능하면 prebuilt 배포만 사용한다.

---

## 2. 기본 배포 프로세스

아래 순서를 기본 규칙으로 따른다.

1. 로컬 코드/콘텐츠 수정
2. 로컬 검증 실행
3. 로컬 빌드 실행
4. 로컬 prebuilt 산출물 생성
5. Vercel에는 prebuilt 결과만 배포

---

## 3. 표준 명령어

```powershell
npm run lint
npm run build
vercel build
vercel deploy --prebuilt --prod
```

설명:
- `npm run lint`: 정적 검사
- `npm run build`: Next.js 로컬 빌드 검증
- `vercel build`: 로컬에서 Vercel 배포 산출물 생성
- `vercel deploy --prebuilt --prod`: 이미 빌드된 결과만 프로덕션에 배포

---

## 4. 금지 사항

아래 방식은 기본적으로 사용하지 않는다.

- `vercel deploy --prod`
- 로컬 빌드 없이 바로 배포
- 원격 빌드 결과만 보고 완료 처리
- 빌드 실패 상태에서 배포 강행

이유:
- Vercel 원격 빌드 비용 발생
- 로컬 검증 없이 배포될 위험 증가
- 문제 원인 추적이 어려워짐

---

## 5. 완료 판단 기준

배포 완료로 간주하려면 아래가 모두 충족되어야 한다.

- 로컬 `npm run lint` 통과
- 로컬 `npm run build` 통과
- 로컬 `vercel build` 통과
- `vercel deploy --prebuilt --prod` 완료
- 배포 URL 확인 완료

---

## 6. 작업 보고 규칙

배포 관련 작업은 아래 순서로 보고한다.

1. 구현
2. 코드 품질 점검
3. 디버그 확인
4. 구현계획 대비 갭 체크
5. 리포트 갱신

배포 리포트에는 최소 아래 내용을 포함한다.

- 로컬 lint 결과
- 로컬 build 결과
- prebuilt 배포 수행 여부
- 최종 배포 URL
- 남은 리스크 또는 수동 확인 항목

---

## 7. 예외

예외적으로 원격 빌드가 필요한 상황이 생기면 사전에 명시하고 진행한다.

예:
- 로컬 환경 제약으로 `vercel build`가 불가능한 경우
- 특정 플랫폼 의존성 때문에 원격 환경 검증이 필요한 경우

이 경우에도 아래를 먼저 남긴다.

- 왜 예외가 필요한지
- 비용/리스크가 무엇인지
- 로컬에서 어떤 검증까지 끝냈는지

---

## 8. 현재 프로젝트 적용 메모

Barracuda 프로젝트는 앞으로 아래 원칙을 기본으로 사용한다.

- 로컬 빌드 우선
- Vercel은 prebuilt 배포만
- 배포 전 검증 필수
- 원격 빌드 배포는 예외 처리
